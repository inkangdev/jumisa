# 배포 / 운영 — Render

Jumisa의 배포 환경·프로세스 기준 문서. (작업 로그가 아니라 **현행 상태를 유지하는 레퍼런스** — 바뀌면 이 파일을 갱신한다.)

관련 파일: `render.yaml`(Blueprint), `backend/Dockerfile`, `backend/src/main/resources/application.yml`
최초 연동 기록: `docs/history/2026-06-14-inkangdev-랜더배포연동.md`

## 구성

DB는 외부(Supabase)라 Render에는 **백엔드 + 프론트 2개 서비스**만 올린다.

| 서비스 | 종류 | 주소 | 비고 |
|---|---|---|---|
| `jumisa` | Static Site (React+Vite) | https://jumisa.onrender.com | **사용자 접속 주소** |
| `jumisa-backend` | Web Service (Docker, Spring Boot) | https://jumisa-backend.onrender.com | API. 사용자가 직접 접속 X |
| (외부) Supabase | Postgres | `...pooler.supabase.com:5432` | 세션 풀러 |

- 리전: `singapore` (무료 중 한국에서 가장 가까움)
- 플랜: 둘 다 `free`

## 자동 배포

- Blueprint가 **`main` 브랜치**에 연결돼 있고, `render.yaml`에 `autoDeploy: false`가 없으므로 **기본값 = 자동 배포 on**.
- 따라서 **`main`에 push(=PR 머지)되면 Render가 자동으로 재빌드·배포**한다.
- `render.yaml`에 경로 필터(buildFilter)가 없어, **변경 파일과 무관하게 두 서비스 모두 재빌드**된다. (프론트만 고쳐도 백엔드도 콜드 재시작 — 무해)

### 일상 배포 흐름

1. feature 브랜치에서 작업 → 커밋·푸시
2. `main`에 머지
3. Render가 자동 빌드 → 수 분 후 반영

## 빌드 방식

- **백엔드** (`backend/Dockerfile`): 멀티스테이지. `eclipse-temurin:21-jdk`로 `./gradlew clean bootJar -x test`(테스트는 Supabase 연결이 필요해 이미지 빌드 시 **제외**) → `21-jre`로 실행. `ENTRYPOINT java -jar app.jar`.
- **프론트** (`render.yaml`): `rootDir: frontend`, `npm ci && npm run build` → `staticPublishPath: ./dist`.

## 라우팅 / 프록시 (프론트 Static Site)

`render.yaml`의 `routes` (위→아래 순서 평가):

1. `/api/*` → `https://jumisa-backend.onrender.com/api/*` **rewrite**
   - 브라우저 입장에선 same-origin이라 **세션 쿠키 인증이 정상 동작**(dev의 vite proxy를 프로덕션에서 재현).
2. `/*` → `/index.html` **rewrite** (SPA 라우팅)

## 환경변수

대시보드에서 직접 입력(시크릿은 `sync: false` → 레포에 안 올라감):

| 키 | 용도 | 비고 |
|---|---|---|
| `SUPABASE_DB_URL` | JDBC URL | 예: `jdbc:postgresql://...pooler.supabase.com:5432/postgres?sslmode=require` |
| `SUPABASE_DB_USER` | DB 사용자 | |
| `SUPABASE_DB_PASSWORD` | DB 비밀번호 | |
| `PORT` | 서버 포트 | **Render가 주입.** `application.yml`이 `server.port: ${PORT:...}`로 우선 사용 |
| `BATCH_SCHEDULER_ENABLED` | 배치 스케줄러 | 기본 비활성(`false`). 필요 시 `render.yaml`에서 주석 해제 |
| `KIS_ENV` / `KIS_APP_KEY` / `KIS_APP_SECRET` / `KIS_ACCOUNT_NO` | KIS API | 시세 적재 가동 시 |

## 헬스체크

- 백엔드 `healthCheckPath: /actuator/health` (200이면 정상). DB 연결 포함.

## 무료플랜 주의 (콜드스타트)

- **미사용 시 서비스가 spin-down** → 첫 요청에서 깨어나는 데 **백엔드 ~40초** 소요.
- 그동안 사용자는 지연을 겪는다. 프론트는 부팅 시 `/api/auth/me` 응답을 기다리므로, **로딩 화면**으로 빈 화면(까만 화면)을 방지함 (`App.tsx`의 `BootingView`, `me()` 12초 타임아웃 — `docs/history/2026-06-14-inkangdev-부팅로딩화면.md`).
- 근본 해소: keep-alive 핑(health 주기 호출)으로 깨워두거나 유료 플랜 상시 가동. (후속 과제)

## 트러블슈팅

- **까만 화면 / 첫 접속 지연**: 십중팔구 백엔드 콜드스타트. `curl https://jumisa-backend.onrender.com/actuator/health` 응답 시간 확인(처음 ~40초면 정상 콜드스타트).
- **`/api/*` 404/CORS**: `render.yaml`의 rewrite 대상 주소가 실제 백엔드 주소와 일치하는지 확인.
- **서비스가 안 잡힘**: Blueprint는 `main` 기준 — `render.yaml`이 다른 브랜치에만 있으면 안 됨.
- **서비스 이름 변경 시**: Render가 새로 만들고 기존 건 자동 삭제 안 함 → 옛 서비스 수동 삭제.
- **Supabase 풀러**: 세션 풀러(5432)는 prepared statement 지원 → 추가 설정 불필요. 트랜잭션 풀러(6543) 사용 시에만 `prepareThreshold: 0` 필요.
