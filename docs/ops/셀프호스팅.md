# 셀프 호스팅 운영 — 내 PC = Render + Supabase (Tailscale Funnel)

24시간 켜둔 Windows PC 한 대에서 jumisa 전체(DB·백엔드·AI·프론트)를 Docker 로 돌리고,
**Tailscale Funnel** 로 무료 고정 HTTPS 주소를 외부에 노출한다. `git push` 하면 자동 재배포된다.

```
[브라우저/폰]  https://<머신>.<테일넷>.ts.net
      │  (Tailscale Funnel, 443)
      ▼
[ web (nginx :8088→80) ]  정적 프론트 + 리버스 프록시
      ├─ /api,/oauth2,/login/oauth2 → [ backend :8080 (Spring Boot) ]
      ├─ /ai                        → [ ai :8000 (FastAPI/Gemini) ]
      └─ /*                         → index.html (SPA)
                                       backend·ai → [ db :5432 (Postgres, 볼륨 pgdata) ]
[ batch (Spring Batch, 비웹) ]  @Scheduled 로 KIS 시세/마스터/재무 자동 적재 ──→ db
```

구성 파일: `docker-compose.yml`, `deploy/`(nginx.conf·web/ai Dockerfile·db-init), `backend/Dockerfile`·`batch/Dockerfile`, `.github/workflows/deploy.yml`.

---

## 0. 사전 준비 (PC 1회)

1. **Docker Desktop** 설치 → 실행 후 확인:
   ```powershell
   docker version
   docker compose version
   ```
2. **Tailscale** 설치·로그인(https://tailscale.com/download). 같은 계정으로 폰에도 깔면 편함.
3. 레포 클론(아직이면): `git clone <repo>` → 폴더로 이동.

---

## 1. 비밀 .env 준비

레포 루트 `.env` 에 값 채움(`deploy/.env.example` 참고). **절대 커밋 금지**(이미 .gitignore 처리됨).

핵심 추가 항목:
```env
POSTGRES_DB=jumisa
POSTGRES_USER=jumisa
POSTGRES_PASSWORD=<강한-비밀번호>
DB_POOL_SIZE=10
FRONTEND_URL=http://localhost:8088   # Funnel 주소 확보 후 교체
KAKAO_CLIENT_ID=...
KAKAO_CLIENT_SECRET=...
GEMINI_API_KEY=...
```
기존 클라우드용 변수(`SUPABASE_DB_URL` 등)는 그대로 둬도 됨 — compose 가 컨테이너 DB 접속을 로컬 값으로 덮어씀.

---

## 2. 기동 & 로컬 확인

```powershell
docker compose up -d --build
docker compose ps            # db/backend/ai/web/batch 가 떠 있는지
docker compose logs -f backend   # 부팅 로그(Ctrl+C 로 빠져나옴)
```
브라우저에서 **http://localhost:8088** → 로그인/배틀/AI질문 동작 확인.

- DB 데이터는 named volume `pgdata` 에 영속. `docker compose down` 해도 보존.
  **`docker compose down -v` 는 볼륨까지 삭제하니 금지.**
- 최초 기동(빈 볼륨) 시 `deploy/db-init` 의 `01-schema.sql`(앱 테이블)·`02-batch.sql`(배치 메타)이 자동 실행됨.

---

## 3. Tailscale Funnel — 공개 주소 노출

1. Tailscale 관리 콘솔(https://login.tailscale.com/admin) → **Settings → Features** 에서
   **HTTPS Certificates** 와 **Funnel** 사용 허용(ACL `nodeAttrs` 에 `funnel`).
2. PC 에서:
   ```powershell
   tailscale funnel 8088
   ```
   출력되는 `https://<머신>.<테일넷>.ts.net` 이 **고정 공개 주소**(폰 LTE 등 외부망에서 접속됨).
   - 상태/해제: `tailscale funnel status` / `tailscale funnel reset`
   - 백그라운드 상시: `tailscale funnel --bg 8088`
3. `.env` 의 `FRONTEND_URL` 을 이 주소로 바꾸고 `docker compose up -d` 재기동.

## 4. 카카오 로그인 도메인 등록

developers.kakao.com → 내 앱 → 카카오 로그인 → **Redirect URI** 에 추가:
```
https://<머신>.<테일넷>.ts.net/login/oauth2/code/kakao
```
플랫폼/사이트 도메인에도 `https://<머신>.<테일넷>.ts.net` 등록.
(nginx 가 `X-Forwarded-Proto: https` 를 넘기므로 백엔드가 redirect_uri 를 https 로 생성한다.)

## 5. 자동배포 — GitHub self-hosted runner (PC 1회)

1. GitHub 레포 → **Settings → Actions → Runners → New self-hosted runner** → **Windows / x64** 안내대로 설치.
2. 상시 실행: `./svc.sh install` 대신 Windows 는 `./config.cmd` 후 **서비스로 등록**(안내에 포함) → PC 재부팅에도 살아있음.
3. 비밀 `.env` 를 러너가 못 받으므로(체크아웃에 없음) **호스트 고정 경로**에 둔다:
   - 예: `C:\jumisa-secrets\.env` (레포의 .env 와 동일 내용).
   - 다른 경로면 러너 PC 환경변수 `JUMISA_ENV_FILE` 로 지정.
   - 워크플로(`.github/workflows/deploy.yml`)가 이 파일을 워크스페이스로 복사 후 `docker compose up`.
4. 이제 `main` 에 push 되면 → 러너가 자동으로 `docker compose up -d --build` 재배포.
   수동 실행: GitHub Actions 탭 → deploy → **Run workflow**.

## 6. DB 데이터 채우기 (stock / 재무) — batch 컨테이너가 자동 적재

stock·financials 는 **batch 컨테이너**가 로컬 DB 에 자동 적재한다(클라우드에서 옮길 필요 없음).
`docker compose up` 에 batch 서비스가 포함돼 **상주 스케줄러**(`BATCH_SCHEDULER_ENABLED=true`, 기본값)로 뜨며,
시간대별로 잡을 자동 실행한다. KIS 접근토큰은 `kis_token` 볼륨(`/app/kis-data/token.json`)에 영속(재발급 제한 회피).

| 잡 | 주기 (Asia/Seoul) |
|---|---|
| 종목 마스터 | 평일 08:00 |
| 시세 스냅샷 | 평일 09~15시 매시 정각 |
| 인트라데이 정리 | 매일 04:30 |
| 재무지표 | 일요일 06:00 |

- DB 접속은 compose 가 자동 구성(`BATCH_DB_URL=jdbc:postgresql://db:5432/jumisa`, user/pw 는 `POSTGRES_*`). 별도 설정 불필요.
- **즉시 1회 적재**(스케줄 시각 안 기다리고): 스케줄러 끈 일회성 컨테이너로 잡 인자 실행 —
  ```powershell
  docker compose run --rm -e BATCH_SCHEDULER_ENABLED=false batch master   # 종목 마스터
  docker compose run --rm -e BATCH_SCHEDULER_ENABLED=false batch price     # 시세 스냅샷
  ```
- 확인: `docker compose exec db psql -U jumisa -d jumisa -c "select count(*) from stock where crno is not null;"` → 2,535 근처.
- 로그: `docker compose logs -f batch`

> member·battle 같은 사용자 데이터는 개발용이라 재가입으로 충분. 굳이 옮기려면 Supabase 에서
> `pg_dump --data-only -t member -t 'battle_*'` → 로컬 `psql ... -f dump.sql`.

## 운영 치트시트

| 작업 | 명령 |
|---|---|
| 기동/재배포 | `docker compose up -d --build` |
| 상태 | `docker compose ps` |
| 로그 | `docker compose logs -f backend` (또는 ai/web/db/batch) |
| 배치 즉시 1회 | `docker compose run --rm -e BATCH_SCHEDULER_ENABLED=false batch master` (또는 price) |
| 중지(데이터 보존) | `docker compose down` |
| 특정 서비스 재시작 | `docker compose restart backend` |
| DB 접속 | `docker compose exec db psql -U jumisa -d jumisa` |
| Funnel 상태 | `tailscale funnel status` |

## 주의

- **`docker compose down -v` 금지** — `pgdata` 볼륨(전체 DB)이 삭제됨.
- `01-schema.sql`·`02-batch.sql` 은 **빈 볼륨 최초 1회만** 실행. 기존 볼륨엔 적용 안 됨(스키마 변경은 수동 마이그레이션).
- 클라우드(Render/Supabase)로 되돌리려면 `.env` 값만 클라우드 주소로 바꾸면 됨(`render.yaml` 보존).
