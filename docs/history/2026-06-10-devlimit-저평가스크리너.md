# 저평가 스크리너 — 점수 계산 방식 — 2026-06-10 (devlimit)

브랜치: `feature/undervalued-stock` (main 기준)

배치로 적재한 데이터를 바탕으로 저평가 점수를 계산해 `undervalue_score`에 적재한다.
기능정의서의 "저평가 점수 산출 및 랭킹" 영역. 점수 가중치는 미결 사항이라 기본값을 제안하고 설정으로 분리한다.

| 테이블 | 건수 | 비고 |
|--------|------|------|

| `stock_financials` | 0 | ❌ 비어있음 — ROE 없음 |

## 선행 작업 — 데이터 적재 3건

### A. `stock_financials` 적재 — stockMasterJob 확장 (작업량 小)

마스터 파일 레이아웃에 **연간재무가 이미 있고 `KisMasterClient`가 파싱까지 하는데 버리고 있다**
(`매출액/영업이익/경상이익/당기순이익/ROE/기준년월` — KisMasterClient.kt 레이아웃 참조).

- `StockMaster`에 재무 필드 추가 → `stock_financials` upsert (`source='master'`, `base_ym`=기준년월)
- API 콜 0건 추가 — 마스터 파일에서 그대로 나옴

### B. `stock_daily` 적재 — closingJob 구현

PER/PBR/EPS/BPS는 마스터 파일에 없고 `inquire-price` 응답에만 있다 (시세적재배치 md에서 확인된 사실).

- 거래가능 종목 순회 → `inquire-price` → `stock_daily` (종가 OHLC + PER/PBR/EPS/BPS/52주) upsert
- 기존 `priceSnapshotJob` 패턴 재사용 (RateLimiter, 종목별 실패 skip)
- ⚠️ 전 종목 ~2,500콜 → **실전(prod) 앱키 필요** (vts는 유량 부적합 — 시세적재배치 md 결론)

### C. `stock.sector` 업종명 보강 — 화면 표기용 (작업량 小)

스크리너 화면이 종목마다 업종(자동차/금융/반도체…)을 표기하는데 현재 `stock.sector`는 전부 null.

- 마스터 파일의 `지수업종대분류` 코드는 **파서가 이미 읽고 있음** (버리는 중) — 코드→업종명 매핑만 추가
- 업종명 매핑은 KIS 업종코드 마스터(`idxcode.mst`) 또는 공식 파서 레포의 분류표 포팅
- ※ v1 점수 계산에는 안 쓰임(화면 표기용). 점수의 섹터 상대평가 전환은 v2.

## 점수 계산 방식 (v1)

### 1단계 — 산출 대상 필터 (Universe)

최신 `base_date` 기준, 전부 만족하는 종목만:

| 조건 | 근거 테이블 | 의미 |
|------|------------|------|
| `security_type = '보통주'` | stock | 우선주/ETF 등 제외 (적재 시 이미 필터됨) |
| `is_tradable = true` | stock | 거래정지·정리매매·관리종목 제외 |
| `market_warning is null` | stock | 투자경고/주의/위험 제외 |
| `per > 0` | stock_daily | 적자 기업 제외 — 이익이 없으면 "이익 대비 싼가" 판단 불가 |
| `pbr > 0` | stock_daily | 자본잠식 제외 |

### 2단계 — 우량 필터 (밸류트랩 제거)

| 조건 | 근거 테이블 | 의미 |
|------|------------|------|
| `roe >= 10` | stock_financials (종목별 최신 base_ym) | 싸기만 한 게 아니라 자본을 굴릴 줄 아는 기업 |
| `roe is not null` | 〃 | 재무 미확인 기업은 산출 제외 |

- 컷 10%는 정석 기준(가치주 체크리스트) 제안값 — **설정 분리** `jumisa.score.filter.min-roe`, 팀 확정 시 변경.

### 3단계 — 점수 산출 (시장 전체 백분위, 0~100)

> 절대값 컷이 아니라 **필터 통과 종목들 사이의 상대 순위**로 점수화한다.
> 백분위는 극단값(PER 수백 배)에 강건하고, 점수가 항상 0~100에 고르게 깔려 랭킹에 적합하다.
> ※ 원래는 업종(섹터) 내 상대평가가 정석이지만 `stock.sector`가 비어있어 v1은 **전체 시장 기준** — 섹터 보강은 확장(아래).

```
per_score = (1 - percent_rank(per)) × 100   -- universe에서 PER 가장 낮으면 100점
pbr_score = (1 - percent_rank(pbr)) × 100   -- universe에서 PBR 가장 낮으면 100점

total_score = w_per × per_score + w_pbr × pbr_score
```

- **가중치 기본 제안: `w_per = 0.6`, `w_pbr = 0.4`** — 이익 기반 지표(PER)가 자산 기반(PBR)보다 시의성이 높다.
- ※ 가중치는 **미결 사항** → 설정 `jumisa.score.weight.per/pbr` 분리, 합=1 검증, 팀 확정 시 값만 교체.

### 4단계 — 랭킹 + 적재

- `total_score` 내림차순 `rank` (1위 = 가장 저평가). 동점 tie-break: PER 낮은 순 → 시총 큰 순.
- `undervalue_score`에 `(base_date, stock_code)` upsert: total_score, rank, per_score, pbr_score.
- `base_date` = 사용한 `stock_daily`의 최신 base_date (산출일이 아닌 데이터 기준일).
- 일별로 쌓이면 기능정의서의 "저평가 랭킹 히스토리(순위 추이)" 근거가 된다.

### 산출 SQL 스케치

```sql
with latest_fin as (                          -- 종목별 최신 재무 (ROE)
    select distinct on (stock_code) stock_code, roe
    from stock_financials
    order by stock_code, base_ym desc
),
universe as (
    select d.stock_code, d.base_date, d.per, d.pbr, d.market_cap_eokwon
    from stock_daily d
    join stock s      on s.stock_code = d.stock_code
    join latest_fin f on f.stock_code = d.stock_code
    where d.base_date = (select max(base_date) from stock_daily)
      and s.security_type = '보통주'
      and s.is_tradable
      and s.market_warning is null
      and d.per > 0 and d.pbr > 0
      and f.roe >= 10                          -- min-roe 설정값 바인딩
),
scored as (
    select stock_code, base_date, per, market_cap_eokwon,
           (1 - percent_rank() over (order by per)) * 100 as per_score,
           (1 - percent_rank() over (order by pbr)) * 100 as pbr_score
    from universe
)
insert into undervalue_score (base_date, stock_code, total_score, rank, per_score, pbr_score)
select base_date, stock_code,
       round((0.6 * per_score + 0.4 * pbr_score)::numeric, 2),
       rank() over (order by 0.6 * per_score + 0.4 * pbr_score desc,
                             per asc, market_cap_eokwon desc),
       round(per_score::numeric, 2), round(pbr_score::numeric, 2)
from scored
on conflict (base_date, stock_code) do update
   set total_score = excluded.total_score, rank = excluded.rank,
       per_score   = excluded.per_score,   pbr_score = excluded.pbr_score;
```

(0.6/0.4와 ROE 컷은 설정값 바인딩으로 교체)

## 확장 여지 (v2 — 이번 범위 아님)

- **배당수익률**: `stock_financials`에 `dps`(주당배당금, 원) 컬럼 추가 + 배당 적재(KIS 예탁원 배당일정 API 또는 공공데이터 연동 선행). 수익률은 저장하지 않고 산출 시 `dps ÷ 최신 종가`로 계산(주가 변동 반영). 코스닥 무배당 종목이 많아 **점수 반영이 아닌 선택 필터**("배당 3% 이상만 보기") + 화면 표시로.
- **부채비율 필터** (≤ 100%): `financeJob`(KIS 재무비율 API) 구현 후 `stock_financials.debt_ratio` 채워지면 우량 필터에 추가.
- **섹터 상대 백분위**: 선행 C(업종명 보강) 완료 후, 점수를 섹터 내 백분위로 전환 (업종별 PER 수준 차이 보정 — 바이오 PER 50 vs 은행 PER 5).
- **EV/EBITDA 필터**: 기능정의서·디자인 초안에 있으나 KIS 미제공 — 부채/현금/EBITDA 데이터 확보 후 산출.
- **ROE 점수 반영**: 필터를 넘어 점수 축으로 (`undervalue_score`에 `roe_score` 컬럼 추가 필요).

## 미결 / 주의

- PER/PBR **가중치 미결** (기능정의서 체크리스트) → 기본 0.6/0.4로 진행, 설정으로 분리.
- ROE 컷(10%)도 팀 확정 대상.
- closingJob은 **실전 앱키** 의존 (현재 `.env`는 `KIS_ENV=vts`).
